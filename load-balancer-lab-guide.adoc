= Hands-on Lab: Load Balancers, WebSocket, gRPC, and Metrics
:toc:
:toclevels: 3
:source-highlighter: rouge

== Learning objectives

By the end of this lab, participants will be able to:

[arabic]
. Understand the difference between Layer 4 and Layer 7 load balancing.
. Observe how traffic is distributed across backend services.
. Compare common load balancing algorithms.
. Understand how load balancing behaves differently for HTTP, WebSocket,
and gRPC traffic.
. See how health checks influence routing decisions.
. Understand sticky sessions and long-lived connections.
. Use Prometheus and Grafana to observe backend traffic distribution.
. Discuss load balancing trade-offs from a software architect’s
perspective.

== 1. Lab Overview

== 2. Architect’s Framing

Before starting the technical work, explain the architectural goal.

A load balancer is not only a traffic distributor. It is an
architectural control point for:

* scalability,
* availability,
* failure isolation,
* traffic routing,
* protocol handling,
* TLS termination,
* observability,
* and sometimes security policy enforcement.

Different protocols create different load balancing concerns.

[width="100%",cols="34%,33%,33%",options="header",]
|===
|Traffic type |Connection behavior |Load balancing concern
|HTTP |Usually request/response |Distribution per request

|WebSocket |Long-lived connection |Distribution per connection

|gRPC |HTTP/2, often long-lived channels |Multiplexed calls over
persistent connections

|Raw TCP |Transport-level stream |No application-level visibility
|===

The key teaching point is this:

____
The load balancer must match the communication pattern, not just the
port number.
____

=== 5.4 app/server.py

....
import os
import time
import threading
from concurrent import futures

from flask import Flask, Response, request
from gevent.pywsgi import WSGIServer
from geventwebsocket.handler import WebSocketHandler
from prometheus_client import Counter, Histogram, generate_latest, CONTENT_TYPE_LATEST

import grpc
import echo_pb2
import echo_pb2_grpc

INSTANCE = os.environ.get("INSTANCE_NAME", "unknown")
HTTP_PORT = int(os.environ.get("HTTP_PORT", "8080"))
GRPC_PORT = int(os.environ.get("GRPC_PORT", "50051"))

app = Flask(__name__)

http_requests = Counter(
    "app_http_requests_total",
    "Total HTTP requests handled by backend instance",
    ["instance", "path"]
)

ws_messages = Counter(
    "app_websocket_messages_total",
    "Total WebSocket messages handled by backend instance",
    ["instance"]
)

grpc_requests = Counter(
    "app_grpc_requests_total",
    "Total gRPC requests handled by backend instance",
    ["instance", "method"]
)

request_latency = Histogram(
    "app_http_request_duration_seconds",
    "HTTP request duration in seconds",
    ["instance", "path"]
)


@app.route("/")
def index():
    with request_latency.labels(INSTANCE, "/").time():
        http_requests.labels(INSTANCE, "/").inc()
        return f"HTTP response from {INSTANCE}\n"


@app.route("/slow")
def slow():
    delay = float(request.args.get("delay", "2"))
    with request_latency.labels(INSTANCE, "/slow").time():
        http_requests.labels(INSTANCE, "/slow").inc()
        time.sleep(delay)
        return f"Slow HTTP response from {INSTANCE} after {delay} seconds\n"


@app.route("/health")
def health():
    return "OK\n"


@app.route("/metrics")
def metrics():
    return Response(generate_latest(), mimetype=CONTENT_TYPE_LATEST)


@app.route("/ws")
def websocket_echo():
    ws = request.environ.get("wsgi.websocket")
    if not ws:
        return "Expected WebSocket request\n", 400

    ws.send(f"Connected to {INSTANCE}")

    while True:
        message = ws.receive()
        if message is None:
            break
        ws_messages.labels(INSTANCE).inc()
        ws.send(f"{INSTANCE} received: {message}")

    return ""


class EchoService(echo_pb2_grpc.EchoServiceServicer):
    def SayHello(self, request, context):
        grpc_requests.labels(INSTANCE, "SayHello").inc()
        return echo_pb2.EchoReply(
            message=f"Hello {request.name} from {INSTANCE}",
            instance=INSTANCE
        )


def start_grpc_server():
    server = grpc.server(futures.ThreadPoolExecutor(max_workers=10))
    echo_pb2_grpc.add_EchoServiceServicer_to_server(EchoService(), server)
    server.add_insecure_port(f"[::]:{GRPC_PORT}")
    server.start()
    print(f"gRPC server for {INSTANCE} listening on {GRPC_PORT}")
    server.wait_for_termination()


if __name__ == "__main__":
    threading.Thread(target=start_grpc_server, daemon=True).start()
    print(f"HTTP/WebSocket server for {INSTANCE} listening on {HTTP_PORT}")
    http_server = WSGIServer(("0.0.0.0", HTTP_PORT), app, handler_class=WebSocketHandler)
    http_server.serve_forever()
....

== 6. NGINX Layer 7 Load Balancer

NGINX will expose:

[width="100%",cols="50%,50%",options="header",]

|Local port |Purpose
|`+8080+` |HTTP and WebSocket traffic

=== 6.1 nginx/nginx.conf

....
events {}

http {
    upstream http_backend {
        server app1:8080;
        server app2:8080;
        server app3:8080;
    }

    upstream grpc_backend {
        server app1:50051;
        server app2:50051;
        server app3:50051;
    }

    server {
        listen 80;

        location / {
            proxy_pass http://http_backend;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        }

        location /ws {
            proxy_pass http://http_backend;
            proxy_http_version 1.1;
            proxy_set_header Upgrade $http_upgrade;
            proxy_set_header Connection "Upgrade";
            proxy_set_header Host $host;
            proxy_read_timeout 3600s;
        }
    }

    server {
        listen 81 http2;

        location / {
            grpc_pass grpc://grpc_backend;
        }
    }
}
....

== 7. HAProxy Layer 4 Load Balancer

HAProxy will expose raw TCP balancing.

[width="100%",cols="50%,50%",options="header",]
|===
|Local port |Purpose
|`+9000+` |TCP load balancing to HTTP backend port 8080
|`+9001+` |TCP load balancing to gRPC backend port 50051
|===

This demonstrates that a Layer 4 load balancer can forward traffic
without understanding HTTP, WebSocket, or gRPC semantics.

=== 7.1 haproxy/haproxy.cfg

....
global
    maxconn 2000

defaults
    mode tcp
    timeout connect 5s
    timeout client 30s
    timeout server 30s

frontend tcp_http_front
    bind *:9000
    default_backend tcp_http_back

backend tcp_http_back
    balance roundrobin
    server app1 app1:8080 check
    server app2 app2:8080 check
    server app3 app3:8080 check

frontend tcp_grpc_front
    bind *:9001
    default_backend tcp_grpc_back

backend tcp_grpc_back
    balance roundrobin
    server app1 app1:50051 check
    server app2 app2:50051 check
    server app3 app3:50051 check

== 12. Experiment 1 — HTTP Round Robin with NGINX

Run:

....
docker compose exec toolbox bash -lc 'for i in {1..12}; do curl -s http://nginx; done'
....

Expected result:

....
HTTP response from app1
HTTP response from app2
HTTP response from app3
HTTP response from app1
...
....

=== Discussion

Round robin is simple and useful when:

* backends have similar capacity,
* requests have similar cost,
* and services are stateless.

=== Architect’s note

Round robin is easy to understand, but it does not know whether one
backend is slower or busier than another.

== 13. Experiment 2 — Weighted Round Robin

Modify the NGINX upstream:

....
upstream http_backend {
    server app1:8080 weight=5;
    server app2:8080 weight=2;
    server app3:8080 weight=1;
}
....

Restart NGINX:

....
docker compose restart nginx
....

Run:

....
docker compose exec toolbox bash -lc 'for i in {1..24}; do curl -s http://nginx; done | sort | uniq -c'
....

Expected result:

`+app1+` should receive more requests than `+app2+`, and `+app2+` more
than `+app3+`.

=== Discussion

Weighted balancing is useful when backend capacity differs.

=== Short example

A migration is in progress. Two old servers remain in service, but one
new server has much more CPU and memory. Weighted load balancing can
shift more traffic to the stronger server.

== 14. Experiment 3 — Least Connections with HAProxy

Modify HAProxy backend:

....
backend tcp_http_back
    balance leastconn
    server app1 app1:8080 check
    server app2 app2:8080 check
    server app3 app3:8080 check
....

Restart HAProxy:

....

....

Generate slow requests in parallel:

....
docker compose exec toolbox bash -lc 'for i in {1..12}; do curl -s "http://haproxy:9000/slow?delay=3" & done; wait'

=== Discussion

Least-connections balancing is useful when requests have different

=== Architect’s note

For long-running requests, equal request counts do not necessarily mean
equal load. Least-connections can be more appropriate than round robin.

== 15. Experiment 4 — Sticky Sessions

Modify NGINX upstream:

....
upstream http_backend {
    ip_hash;
    server app1:8080;
    server app2:8080;
    server app3:8080;
}
....

Restart:

....
docker compose restart nginx
....

Run from the same machine:

....
docker compose exec toolbox bash -lc 'for i in {1..10}; do curl -s http://nginx; done'
....

Expected result:

Requests from the same client usually go to the same backend.

=== Discussion

Sticky sessions are useful for legacy applications that keep session
state in local memory.

=== Architectural trade-off

Sticky sessions reduce elasticity. If one backend fails, user sessions
may be disrupted unless session state is externalized.

A more scalable architecture stores session state outside the
application instance, for example in Redis or a database.

== 16. Experiment 5 — WebSocket Load Balancing

WebSocket begins as an HTTP request and then upgrades the connection to
a long-lived bidirectional channel.

This changes the load balancing behavior.

For normal HTTP, a load balancer can distribute every request
independently.

For WebSocket, the load balancer usually chooses a backend when the
connection is established. After that, messages stay on the same
connection and therefore the same backend.

=== 16.1 Test WebSocket from the toolbox container

Run the WebSocket client from the toolbox container:

....
docker compose exec toolbox websocat ws://nginx/ws
....

Type:

....
hello
....

Expected response:

....
Connected to app1
app1 received: hello
....

Open two or three terminals and connect again:

....
docker compose exec toolbox websocat ws://nginx/ws
....

You should see different backend instances across different WebSocket
connections.

=== Discussion

For WebSocket systems, distribution is usually per connection, not per
message.

This matters because WebSocket connections can remain open for minutes
or hours.

=== Short example

A trading dashboard opens one WebSocket connection and receives live
market data. If 10,000 users connect, the load balancer distributes
10,000 long-lived connections. It does not rebalance every message.

=== Architect’s note

WebSocket architectures need careful planning for:

* connection count,
* backend memory usage,
* idle timeouts,
* reconnection behavior,
* sticky routing,
* and graceful shutdown.

== 17. Experiment 6 — gRPC Load Balancing

gRPC commonly runs over HTTP/2. HTTP/2 allows multiple logical requests
over the same long-lived connection.

This creates an important load balancing lesson:

____
If a client opens one persistent HTTP/2 connection, many gRPC calls may
continue to use the same backend.
____

This can surprise teams expecting request-by-request distribution.

=== 17.1 Test gRPC from the toolbox container

From the toolbox container, use the mounted proto file at
/workspace/app:

....
docker compose exec toolbox grpcurl -plaintext \
  -import-path /workspace/app \
  -proto echo.proto \
  -d '{"name":"Architect"}' \
  nginx:81 echo.EchoService/SayHello
....

Expected response:

....
{
  "message": "Hello Architect from app1",
  "instance": "app1"
}
....

Run it multiple times:

....
docker compose exec toolbox bash -lc '
for i in {1..12}; do
  grpcurl -plaintext \
    -import-path /workspace/app \
    -proto echo.proto \
    -d "{\"name\":\"Architect\"}" \
    nginx:81 echo.EchoService/SayHello
done
'
....

Depending on connection behavior, calls may distribute across backends
or reuse connections.

=== 17.2 Test gRPC through HAProxy Layer 4

....
docker compose exec toolbox grpcurl -plaintext \
  -import-path /workspace/app \
  -proto echo.proto \
  -d '{"name":"L4"}' \
  haproxy:9001 echo.EchoService/SayHello
....

=== Discussion

Layer 4 balancing can forward gRPC because it is TCP traffic, but it
does not understand gRPC methods, metadata, or application-level health.

Layer 7 balancing can make more application-aware decisions, depending
on the load balancer and configuration.

=== Architect’s note

For gRPC-heavy systems, architects should consider:

* HTTP/2 connection reuse,
* client-side load balancing,
* service discovery,
* keepalive settings,
* retries and deadlines,
* observability,
* and whether balancing happens at the connection level or request
level.

== 18. Experiment 7 — Health Checks and Failure Behavior

Stop one backend:

....

....

....

....

=== Discussion

Questions for participants:

[arabic]
. How quickly does the load balancer stop routing traffic to the failed
backend?
. Are failures visible to the client?
. What happens to existing WebSocket connections when a backend stops?
. What happens to gRPC clients with long-lived channels?

=== Architect’s note

Health checks are not just operational details. They define what the
platform considers healthy.

A shallow health check may say “OK” even when the database is
unavailable. A deep health check may be more accurate but may also
create additional load or false failures.

== 21. Experiment 10 — Compare HTTP, WebSocket, and gRPC

Use the metrics dashboard while generating traffic.

=== HTTP traffic

....
docker compose exec toolbox bash -lc 'for i in {1..100}; do curl -s http://nginx > /dev/null; done'
....

* Many short requests
* Distribution visible per request

=== WebSocket traffic

Open multiple WebSocket clients from separate terminals:

....
docker compose exec toolbox websocat ws://nginx/ws
....

Send several messages through one connection.

Expected behavior:

* One backend handles all messages for that connection
* Distribution happens when new connections are established

=== gRPC traffic

Run repeated gRPC calls:

....
docker compose exec toolbox bash -lc '
for i in {1..50}; do
  grpcurl -plaintext \
    -import-path /workspace/app \
    -proto echo.proto \
    -d "{\"name\":\"metrics\"}" \
    nginx:81 echo.EchoService/SayHello > /dev/null
done
'
....

Expected behavior:

* Distribution depends heavily on connection reuse
* Persistent channels may reduce request-level balancing

== 22. Optional Advanced Experiment — Add Artificial Latency

=== Discussion

== 25. Suggested Instructor Narrative

=== Opening

In the lecture, load balancers were introduced as Layer 4 and Layer 7
components. In this lab, participants will experience why that
distinction matters.

A load balancer is not just a box that spreads requests. It makes
assumptions about protocols, connections, health, and failure.

=== During HTTP tests

Point out that HTTP makes distribution easy to observe because each
request is short. This is the simplest mental model of load balancing.

=== During WebSocket tests

Emphasize that WebSocket changes the game. After upgrade, the connection
stays open. The balancing decision is made at connection time, not for
every message.

=== During gRPC tests

Explain that gRPC usually uses HTTP/2. HTTP/2 multiplexes many calls
over a connection. This means the connection strategy can dominate the
balancing behavior.

=== During metrics tests

Show that observability turns architecture from theory into evidence. It
allows architects to validate assumptions about load distribution,
latency, and failure behavior.

=== Closing

The main conclusion is simple:

____
Load balancing is not one pattern. It is a family of patterns shaped by
protocol, connection lifetime, health model, and operational visibility.
____

== 26. Common Troubleshooting

=== NGINX does not start

Check configuration:

....
docker compose logs nginx
....

Common causes:

* missing semicolon in `+nginx.conf+`,
* wrong upstream name,
* wrong mounted path.

=== HAProxy does not start

Check logs:

....
docker compose logs haproxy
....

Common causes:

* invalid indentation,
* unsupported directive,
* backend name typo.

=== Prometheus has no targets

Open:

....
http://localhost:9090/targets
....

Check whether `+app1+`, `+app2+`, and `+app3+` are UP.

=== Grafana has no data

Check:

[arabic]
. Prometheus datasource URL is `+http://prometheus:9090+`.
. Prometheus targets are UP.
. Traffic has been generated after startup.

=== grpcurl fails

Check:

[arabic]
. NGINX exposes port `+8081+`.
. The gRPC server inside each app listens on `+50051+`.
. The command uses `+-plaintext+`.
. The proto path points to /workspace/app when grpcurl runs inside the
toolbox container.

=== WebSocket client cannot connect

Check:

[arabic]
. NGINX `+/ws+` location includes `+Upgrade+` and `+Connection+`
headers.
. The backend uses `+gevent-websocket+`.
. The toolbox WebSocket URL is ws://nginx/ws. If a host-installed client
is used instead, use ws://localhost:8080/ws.

== 28. Architect’s Final Takeaways

[arabic]
. HTTP load balancing is often request-oriented.
. WebSocket load balancing is connection-oriented.
. gRPC load balancing is strongly affected by HTTP/2 connection reuse.
. Layer 4 load balancers are fast and protocol-agnostic, but less
application-aware.
. Layer 7 load balancers can make smarter routing decisions, but require
protocol understanding.
. Algorithms must match workload behavior.
. Health checks define what “available” means.
. Metrics are necessary to validate architecture assumptions.
. Sticky routing is sometimes useful, but it creates scalability
trade-offs.
. The architect’s job is to connect protocol behavior to quality
attributes: performance, availability, scalability, operability, and
cost.
